# 🛠 쿠폰 동시성 발급 | 2026.04.27

---

### ❗ 문제 상황

- 선착순 쿠폰 발급 기능에서 동시 요청이 발생하는 상황을 테스트해야 했음
- 다수의 사용자가 동시에 요청할 경우
    - 중복 발급
    - 초과 발급
    - 테스트 결과 불일치 (사일런트 실패)
      등의 문제가 발생할 수 있음
- AI 코드 리뷰에서 동시성 테스트 및 검증 방식에 대해 다수의 피드백이 제시됨

---

### 🔍 원인

#### 1. 동시성 테스트 구성 문제

- `CyclicBarrier` 사용 시 `barrier.await()` 호출 누락
- `BrokenBarrierException` catch는 존재하지만 실제 발생 경로 없음 → 컴파일 에러 발생

---

#### 2. 사일런트 실패 발생

```
} catch (InterruptedException | BrokenBarrierException e) {
    Thread.currentThread().interrupt();
}
```
- 실패 카운트 누락으로 일부 스레드가 성공/실패 어디에도 집계되지 않음
- 결과적으로 테스트 수치 불일치 발생 (예: 150명 중 일부 누락)

#### 3. 동시성 테스트의 비결정성
- 스레드 실행 순서, 커넥션 풀 상태 등에 따라 결과가 달라짐
- 동일 코드라도 실행마다 결과가 달라지는 현상 발생

#### 4. "실패해야 하는 테스트"에 대한 해석 문제
- 발제 가이드에서 “이 시점의 테스트는 실패해야 정상”이라는 문구 존재
 - 하지만 실제 구현은 이미
- Redis 락
- DB 조건부 UPDATE
- 두 단계 보호가 적용된 상태

→ 의도적으로 실패 테스트를 만들기 어려운 구조

### 🛠 해결 방법
#### 1. CyclicBarrier 정상 적용
```
   executorService.submit(() -> {
   try {
   barrier.await(); // 동시 시작 보장
   // ...
```
- 모든 스레드가 동일 시점에 시작하도록 구성
#### 2. 사일런트 실패 방지
```
   } catch (InterruptedException | BrokenBarrierException e) {
   failCount.incrementAndGet();
   Thread.currentThread().interrupt();
   }
```
- 모든 예외 경로에서 반드시 실패 카운트 처리

#### 3. 테스트 설계 방향 수정
- “실패 테스트”를 억지로 만들지 않고 현재 구조가 정상적으로 동작하는지 검증하는 방향으로 전환

#### 4. AI 리뷰 대응 전략
- 모든 피드백을 한 PR에 반영하지 않고
- 필수 항목만 반영
- 나머지는 별도 이슈로 분리

### ✅ 결과
- 동시성 테스트 안정적으로 실행 가능
- 모든 요청이 성공 또는 실패로 정확히 집계됨
- 쿠폰 발급 수량이 정확히 제한 내에서 유지됨
- 테스트 결과의 일관성 확보

### 📌 배운 점
- 동시성 테스트는 본질적으로 비결정적이므로 모든 경로에서 결과를 집계해야 한다
- CyclicBarrier는 선언 + 호출 + 예외 처리 세트가 모두 필요하다
- 테스트 설계 시 “실패를 만드는 것”보다 “정상 동작을 검증하는 것”이 더 중요할 수 있다
- AI 리뷰는 참고 자료일 뿐, 반드시 요구사항과 대조하여 판단해야 한다
