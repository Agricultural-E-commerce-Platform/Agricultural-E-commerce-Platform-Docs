## 🛠 주문 생성 시 재고 차감 동시성 문제 | 26.04.24

### 🚨 핵심 요약
👉 **주문 동시 요청 시 재고 음수 발생 → Redis 다중 락으로 해결**

---

### ❗ 문제 상황

여러 사용자가 동시에 동일 상품을 주문할 경우  
재고 검증과 차감이 분리되어 실행되면서 **재고 음수 및 초과 주문 문제 발생 가능**

#### 기존 흐름
1. 장바구니 조회
2. 재고 확인
3. 주문 생성
4. 재고 차감
5. 장바구니 삭제

→ 동시에 요청 시 **모든 요청이 동일 재고 기준으로 검증 통과**

---

### 🔍 원인

- 재고 확인 + 차감이 **원자적으로 처리되지 않음**
- 트랜잭션 내에서도 **Race Condition 발생**
- 주문 / 재고 / 장바구니 로직이 묶여 있어 **부분 커밋 위험 존재**

---

### 🛠 해결 방법

#### 1. Redis 기반 분산 락 적용

- 상품 단위 Lock (`product:stock:{id}`)
- 다중 상품 주문 시 **정렬 후 락 획득 → 데드락 방지**

```
.map(productId -> "product:stock:" + productId)
.sorted()
```

#### 2. 락 획득 이후에만 주문 로직 실행
```
lockService.executeWithLocks(...)
```
#### 3. 설계 개선
- 재고 차감 로직을 동시성 제어 범위 안으로 이동

#### 🤔 왜 Redis Lock을 선택했는가
- DB 락은 단일 인스턴스 환경에 적합
- 현재 구조는 확장 가능성을 고려해야 했음
- 애플리케이션 레벨에서 분산 환경 제어 필요

👉 따라서 Redis 기반 분산 락 선택

### ✅ 결과
- 동시 요청 시 재고 음수 문제 해결
- 상품 단위 락으로 불필요한 성능 저하 방지
- 다중 상품 주문에서도 데드락 없이 안정 처리

### 💡 배운 점
- 재고 검증과 차감은 반드시 원자적으로 처리해야 한다
- 동시성 문제는 코드가 아니라 설계(락, 트랜잭션)로 해결해야 한다
- 락은 최소 범위(상품 단위)로 설계해야 성능을 유지할 수 있다
- 동시성 이슈는 멀티스레드 테스트로만 검증 가능하다
