# 🤝 팀 협업 정책 (Team Collaboration Policy)

## 🎯 프로젝트 지향점

- 단순 CRUD 구현을 넘어 **동시성 제어, 캐싱, 검색 성능 개선을 중심으로 한 백엔드 시스템 구현**
- Redis Lock과 DB 조건 검증 기반 동시성 제어를 통해 **실제 서비스 환경에서 발생할 수 있는 데이터 정합성 문제 해결**
- 문제 해결 과정을 문서화하여 **지식 공유 및 재사용 가능하도록 관리**

---

## 👥 역할 및 책임

| 이름 | 역할 | 담당 도메인 | 핵심 역할 |
|------|------|-------------|----------|
| 정지훈 | 팀장 / DevOps | auth / user / infra | CI/CD, 인증, 공통 모듈 |
| 정은지 | 부팀장 | product / search | 상품, 검색, 캐싱 |
| 김예은 | QA / 서기 | order / cart | 주문, 장바구니 |
| 이중현 | QA 총괄 / DB | coupon / admin | 쿠폰, 동시성 |

---

## 📚 문서화 전략

- 코드와 문서를 분리하여 GitHub Docs 레포에서 관리
- 주요 문서:
  - 서비스 정책
  - ERD
  - API 명세
  - 트러블슈팅
  - 회의록

---

## 🔀 Git 협업 전략

### 브랜치 전략

```text
main ← dev ← feature
```

- PR 기반 협업 진행
- 1명 이상 승인 후 머지
- Squash Merge 사용

---

## 📝 엔지니어링 기준

- Controller-Service-Repository 구조 사용
- Service 계층에서만 `@Transactional` 사용
- Enum을 활용하여 상태값 관리

---

## 🧪 테스트 전략

- Postman 기반 1차 테스트
- JUnit + Mockito 단위 테스트
- SpringBootTest 통합 테스트
- 교차 테스트 방식 적용

### 동시성 검증

- 쿠폰 발급 → Redis 분산 락
- 상품 재고 차감 → Redis 분산 락 + DB 조건부 UPDATE
- 타임세일 상태 변경 → JPA 비관적 락

---

## 🚨 협업 규칙

- 문제 발생 시 30분 내 해결 시도 후 공유
- 이슈 기반 작업 및 PR 연동
- 모든 기능은 테스트 통과 후 완료
