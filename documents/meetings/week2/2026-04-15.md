## 💡 기술 선택 고민 (특가 상품 기능)

특가 상품 기능 구현 과정에서, **정해진 시각에 상태가 정확하게 변경되어야 한다**는 요구사항을 기반으로 다음 네 가지 방식을 비교 검토


### 1. 조회 시점 계산 방식

상품 조회 시 현재 시간과 `saleStartTime`, `saleEndTime`을 비교하여 상태를 동적으로 판단하는 방식

#### 장점
- 별도의 스케줄러가 필요 없음
- 서버가 여러 대여도 동일한 로직으로 처리 가능 → 구조 단순

#### 단점
- `BETWEEN` 조건으로 인해 인덱스 활용이 어려워 대량 데이터에서 성능 저하
- 서버 시간과 DB 시간 불일치 시 상태 불일치 발생 가능
- 상태값이 DB에 저장되지 않아 다른 도메인에서 재사용 어려움
- “상태 변화”가 아닌 “조회 시 계산” 방식 → 도메인 의미 약화


### 2. Redis 기반 예약 키 + 폴링 방식

Redis에 시작/종료 시간을 저장하고 주기적으로 확인하여 상태를 변경하는 방식

#### 장점
- 메모리 기반으로 빠른 처리
- 재고, 분산락 등과 자연스럽게 연계 가능

#### 단점
- 결국 폴링 구조 필요
- 정확한 시각 보장이 어려움
- 주기적 조회로 인해 불필요한 리소스 사용


### 3. @Scheduled 기반 폴링 스케줄러

Spring의 `@Scheduled`를 사용해 일정 주기마다 전체 상품을 조회하여 상태 변경

#### 장점
- 구현이 간단함

#### 단점
- 전체 상품 반복 조회 → 비효율
- 서버가 여러 대일 경우 중복 실행 위험
- “정확한 시각 실행” 요구사항과 부합하지 않음


### 4. Quartz 기반 예약 실행 방식

상품 등록 시점에 시작/종료 Job을 예약하여 해당 시각에 상태를 변경하는 방식

#### 장점
- 폴링 없이 정확한 시각에 실행 가능
- 상태를 실제 값으로 관리 → 도메인 명확성 확보
- 불필요한 조회 제거 → 성능 효율성
- 기능별 책임 분리 용이

## 🎯 최종 선택: Quartz

특가 상품의 핵심 요구사항은 **“정해진 시작/종료 시각에 정확하게 상태가 변경되는 것”**

이를 만족하기 위해 폴링 방식이 아닌 **예약 실행 방식(Quartz)**을 선택


### 선택 이유

#### 1. 정확한 실행 보장 (No Polling)
- 특정 시각에 Job을 직접 예약하여 실행 가능

#### 2. 도메인 상태의 명확성
- `READY → ON_SALE → SALE_ENDED`
- 상태 변화가 명확하게 관리됨

#### 3. 책임 분리 (Separation of Concerns)
- 예약 실행 → Quartz
- 재고 및 동시성 처리 → Redis

#### 4. 성능 최적화
- 불필요한 반복 조회 제거


#### Redis 활용 전략

Redis는 예약 실행이 아닌 **동시성 제어 및 재고 처리**에 활용

- 메모리 기반으로 빠른 처리 속도
- 분산 환경에서 락 처리 가능
- 선착순 구매 / 중복 구매 방지에 적합


## 결론

- **Quartz** → 시간 기반 상태 변경
- **Redis** → 실시간 동시성 제어
