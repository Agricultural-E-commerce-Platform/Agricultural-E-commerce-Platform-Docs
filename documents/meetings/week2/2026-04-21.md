### 피드백 및 추가 설명
* **서비스 분리 및 멀티 인스턴스 도입 목적**

    1. 서비스 안정성 확보
       → 한쪽 서버 장애 발생 시에도 다른 인스턴스로 대응 가능

    2. 서비스 과부하 방지
       → 트래픽 분산을 통해 서버 부하 감소

  → 위 두 가지 이유만으로도 인스턴스 분리는 충분히 타당함

    * 추가 목적:
      → 무중단 배포(서비스 중단 없이 배포)를 위해 멀티 인스턴스 활용

    * 전체 흐름:
      → 멀티 인스턴스 도입 → 데이터 정합성 문제 발생 → 분산락 등으로 동시성 제어

* **멀티 인스턴스 구현 방식**

    * 서버 2개 이상 구성 + 로드밸런서(또는 리버스 프록시) 설정

    * 구성 요소 예시:

        * Nginx (리버스 프록시 + 로드밸런서 역할 가능)
        * HAProxy (로드밸런싱 특화)
        * AWS ELB (Managed Load Balancer)
        * AWS API Gateway (API 관리 + 일부 라우팅, LB와는 역할 다름)

    * 로드밸런싱 방식:

        * Round Robin
        * Weighted Round Robin
        * (추가) Least Connections 등도 존재 → 필요 시 학습

* **Docker 헬스체크 개선 필요**

    * 기존: `docker ps`로 컨테이너 실행 여부만 확인
    * 개선: 컨테이너 실행 후 일정 시간 대기(sleep) + `/actuator/health` 호출
      → 애플리케이션 레벨에서 정상 동작 여부 확인
